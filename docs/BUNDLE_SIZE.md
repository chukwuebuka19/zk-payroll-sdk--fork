# Bundle Size Budget & Optimization Plan

Addresses [Issue #53](https://github.com/zkpayroll/zk-payroll-sdk/issues/53).

---

## Baseline Measurements

Measurements reflect the compiled CommonJS output (`dist/`) of `@zk-payroll/core@0.1.0` before tree-shaking, as generated by `tsc`. Browser bundle sizes are estimated via `esbuild` with minification and gzip compression.

| Module (import path)         | Raw (min) | Gzip   | Dominant dependency        |
|------------------------------|-----------|--------|----------------------------|
| Full SDK (`@zk-payroll/core`)| ~2.8 MB   | ~820 KB| snarkjs + @stellar/stellar-sdk |
| Proof generation only        | ~1.5 MB   | ~460 KB| snarkjs (WASM bundled)     |
| Contract adapters only       | ~820 KB   | ~250 KB| @stellar/stellar-sdk       |
| Batch utilities only         | ~4 KB     | ~2 KB  | none (pure TypeScript)     |
| Logging hooks only           | ~2 KB     | ~1 KB  | none (pure TypeScript)     |

> **Note:** Run `npm run bundle:measure` after building (`npm run build`) to get
> exact per-file sizes for your local build.

## Top Bundle Contributors

| Dependency              | Estimated size (gzip) | Reason for inclusion                      |
|-------------------------|-----------------------|-------------------------------------------|
| `snarkjs`               | ~370 KB               | Groth16 proof generation (includes WASM)  |
| `@stellar/stellar-sdk`  | ~250 KB               | Soroban RPC, XDR encoding, Keypair        |
| `stellar-sdk`           | ~130 KB               | Older SDK bundled alongside the above     |
| `axios`                 | ~13 KB                | Artifact downloads (.wasm / .zkey)        |

`snarkjs` is the single largest contributor. It ships a compiled WASM binary for the
BN128 curve that cannot be eliminated through tree-shaking.

---

## Bundle Size Target

| Scope                                    | Target (gzip) | Rationale                                      |
|------------------------------------------|---------------|------------------------------------------------|
| Core SDK (proof gen + contract adapters) | ≤ 900 KB      | Acceptable for a Node.js/server-side payroll SDK |
| Browser-facing utilities (batch, logging)| ≤ 10 KB       | Pure TS modules with zero heavy deps           |
| Future: browser-only entry point         | ≤ 50 KB       | ZK verification only, no proof generation      |

## Identified Optimization Opportunities

### 1. Separate the proof generation entry point

`snarkjs` is only needed on the **server side** (or in a Web Worker). Split the package
into two entry points:

```jsonc
// package.json
"exports": {
  ".":           "./dist/index.js",          // full SDK
  "./browser":   "./dist/browser/index.js",  // lightweight browser bundle
  "./proof-gen": "./dist/crypto/index.js"    // server-side proof tools only
}
```

Browser apps that only need contract interaction or batch building can import
`@zk-payroll/core/browser` and avoid pulling in snarkjs entirely.

### 2. Replace axios with the Fetch API

`axios` is used solely for downloading `.wasm` and `.zkey` circuit artifacts. Modern
browsers and Node.js 18+ ship `fetch` natively:

```typescript
// Before (~50 KB dependency)
const res = await axios.get(url, { responseType: "arraybuffer" });

// After (zero dependency)
const res = await fetch(url);
const buf = await res.arrayBuffer();
```

Eliminating `axios` saves ~13 KB gzip and reduces the dependency surface.

### 3. Deduplicate stellar-sdk and @stellar/stellar-sdk

Both `stellar-sdk` and `@stellar/stellar-sdk` are declared as dependencies. The former
is a legacy alias for the latter. Remove `stellar-sdk` from `dependencies` and import
only from `@stellar/stellar-sdk`:

```diff
- "stellar-sdk": "^11.0.0",
  "@stellar/stellar-sdk": "^14.5.0",
```

Estimated saving: ~130 KB gzip.

### 4. Lazy-load snarkjs

When using the preloading API (`proofGenerator.preload()`), snarkjs can be dynamically
imported so it does not block the initial module load:

```typescript
// Dynamic import — does not affect initial bundle
const { groth16 } = await import("snarkjs");
```

This keeps the main thread responsive while the WASM module is being compiled by the JS
engine in the background.

### 5. Externalize heavy dependencies for library consumers

When downstream apps bundle the SDK with webpack/esbuild/Rollup, they will double-bundle
`snarkjs` and `@stellar/stellar-sdk` unless those packages are declared as `peerDependencies`
or `externals`. For browser-targeted SDK builds, move these to `peerDependencies`.

## How to Measure

After building the SDK (`npm run build -w packages/core`), run the built-in script:

```bash
npm run bundle:measure -w packages/core
```

This reports the size of every file in `packages/core/dist/` plus a gzip-compressed
estimate. To get a full browser bundle estimate, use:

```bash
npx esbuild packages/core/dist/index.js \
  --bundle --minify --platform=browser \
  --outfile=/tmp/sdk-bundle.js && \
  gzip -c /tmp/sdk-bundle.js | wc -c
```

---

## Tracking

Bundle size should be checked on every release. The CI pipeline can enforce the budget
by adding a `bundlesize` step that fails if gzip size exceeds the target values above.
